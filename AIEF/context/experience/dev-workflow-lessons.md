# 功能开发完整流程规范

> 经验教训沉淀 - 2026-03-13
> 来源：Sidecar L2 开发中文档更新滞后问题

## 核心教训

**代码和文档必须同步交付。文档不是可选项，是交付的必要组成部分。**

## 错误案例分析

### 错误做法（本次犯的错误）

**场景**: Markdown→PDF 导出功能实现

**实际流程**:
```
代码实现 → 测试通过 → PR #17 → 合并 → （忘了文档）→ PR #18 补文档
```

**问题**:
1. 代码和文档不同步，间隔数小时
2. 需要额外的 PR 来修复（浪费审查资源）
3. 项目状态文档长期显示过时信息
4. 历史记录混乱（一个功能两个 PR）

**根本原因**: 没有把"更新文档"纳入完成定义（Definition of Done）

---

## 正确流程

```
代码实现 → 测试验证 → 文档更新 → PR（代码+文档）→ 审查 → 合并
```

### 各阶段详细要求

#### 1. 代码实现
- 遵循 TDD（先写失败测试，再写实现）
- 通过 lint 和 type check
- 无调试代码或 TODO 遗留

#### 2. 测试验证
- 单元测试全部通过
- 集成测试（如有）
- UI 构建成功（前端改动）
- 手动验证关键路径

#### 3. 文档更新 ⚠️ **最容易遗漏**

**必须检查**:

| 文档类型 | 位置 | 检查项 |
|---------|------|--------|
| 实现计划 | `AIEF/docs/plans/YYYY-MM-DD-*.md` | 标记任务完成、添加完成摘要 |
| 项目状态 | `AIEF/docs/plans/project-status-and-next-steps.md` | 更新功能列表、标记完成状态 |
| API 契约 | `ui/design/contracts/*.md` | 更新请求/响应格式 |
| 设计文档 | `openspec/changes/*/design.md` | 更新实现状态 |

**文档更新模板**:
```markdown
## 完成功能：[功能名称]

- **状态**: ✅ 已完成
- **完成日期**: YYYY-MM-DD
- **相关 PR**: #XXX
- **测试状态**: X passed

### 实现内容
- 功能点 1
- 功能点 2

### 已知限制
- [限制说明]
```

#### 4. PR 准备
- `git diff --stat` 检查改动范围
- 确保提交信息规范
- **PR 必须同时包含代码和文档**

#### 5. 提交 PR
- PR 描述清晰说明改动
- 关联相关 Issue
- 使用检查清单模板

#### 6. 审查与合并
- CI 检查通过
- 审查者确认文档已更新
- 合并到主分支

## 防错机制

### 1. PR 模板强制检查

创建 `.github/pull_request_template.md`:
```markdown
## 检查清单

### 代码
- [ ] 功能实现完成
- [ ] 单元测试通过
- [ ] 无 lint/type 错误

### 文档 ⚠️
- [ ] 实现计划文档已更新
- [ ] 项目状态文档已更新（如需要）
- [ ] API 文档已更新（如接口变更）
```

### 2. 提交前自检问题

在准备 PR 时问自己：
1. 这个功能修改了哪些行为？→ 需要更新用户文档吗？
2. 接口有变化吗？→ 需要更新 API 契约吗？
3. 这是一个新功能吗？→ 需要更新项目状态文档吗？
4. 实现计划中有这个任务吗？→ 需要标记完成吗？

### 3. 代码审查时的文档检查

审查清单必须包含：
- [ ] 文档是否随代码同步更新？
- [ ] 新功能是否有文档说明？
- [ ] 接口变更是否更新了契约文档？
