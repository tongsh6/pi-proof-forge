## 0. 执行摘要
### 0.1 项目要解决的问题
- 简历定制成本高、质量不稳定；生成内容常“像但不真”（空话/泛化/缺证据）。
- 面对不同公司/岗位，缺少可复用的“事实资产”与可解释的匹配闭环。
- 投递后缺少数据回流，无法系统性提升面试率。
- **现有 AI 工具普遍假设“知识已结构化”，但真实用户往往只有多年碎片化材料。**

### 0.2 核心方案
- 将工作库拆为两层：**日志层（原始材料）** + **证据层（经历卡，results+artifacts 强约束）**
- 以证据层为唯一事实源：**召回 → 匹配评分（可解释）→ 简历拼装（模板化）→ 受控改写（不改事实）→ 质量评测 → 追踪回流**

### 0.3 关键差异化
- **Evidence-first**：每条简历表达都能追溯到经历卡与证据。
- **可解释匹配**：不仅给分数，还给“缺口清单 + 补证据任务”。
- **可复现流水线**：同一 JD + 同一证据库 → 输出稳定可版本化（便于 A/B 投递）。

### 0.4 核心 AI 主张（开源定位）
**Most AI generation tools assume your knowledge is already structured. This project doesn't.**

项目的核心价值不是“生成更漂亮的简历”，而是补上长期缺失的前置步骤：

```
[多年工作材料（非结构化）]
    ↓ 证据提炼（结构化、可验证）
[证据层经历卡]
    ↓ 可解释匹配 + 事实保真生成
[目标文档：简历/提案/述职/...]
    ↓ 评测反馈
[补证据任务队列]
```

这意味着项目的开源价值在于：
- **Evidence Extraction**：从原始材料提炼结构化证据（最容易被忽略的真实难点）。
- **Fact-Preserving Generation**：任何输出必须可追溯至证据字段，禁止“合理推断”。
- **Evidence Gap Feedback**：生成的缺口反驱证据补全，形成闭环。

求职场景只是第一个参考实现，框架本身适用于任何“原始材料 → 可信文档”的生成场景。

## 1. 背景与需求
### 1.1 现状痛点
- 材料碎片化：周报、排障、复盘、PR、监控截图、压测报告等难以复用。
- “项目列表式简历”无说服力：缺指标、缺证据、缺岗位语言。
- 传统工具多围绕“现有简历文本”做关键词优化，难保证真实性与可验证性。

### 1.2 用户目标
- 2026 年 3 月求职：在可控成本下，提高命中率/面试率，并沉淀长期可复用的职业资产。

### 1.3 问题抽象（从求职工具到开源框架）
将问题从“写简历”抽象为：

```
[原始材料] → [结构化证据] → [可解释匹配与生成] → [质量评测与反馈]
```

**关键差异**：不假设用户已有整理好的知识资产，把“证据提炼”作为系统第一步。

## 2. 目标与非目标
### 2.1 项目目标（可量化）
1. 建立证据层：至少 **20 张**可用经历卡（results+artifacts 完整）。
2. 任意 JD 输入后，**15 分钟内**输出：
   - 匹配报告（可解释分项得分 + 缺口）
   - 两版 1 页定制简历（A：偏架构；B：偏交付/Owner）
3. 简历质量自动评测：
   - must-have 关键词覆盖率
   - 量化占比（含数字 bullets 比例）
   - 空话/重复度
   - 篇幅/可读性
4. 投递追踪：至少 **20 次投递**形成版本-结果数据，驱动权重与模板迭代。

### 2.2 非目标（MVP 不做或弱做）
- 不做“全自动投递轰炸”（噪音、合规与平台风控风险）。
- 不做超出终版 9 页信息架构范围的复杂平台扩张（先落地终版桌面端核心能力）。
- 不做复杂多 Agent 辩论（先保证确定性与评测可落地）。

## 3. 用户画像与使用场景
### 3.1 核心用户
- 单人求职者（优先工程师/技术岗），具有可追溯工作材料。

### 3.2 核心场景
1. 录入：从日志/材料提炼经历卡并补齐指标与证据。
2. 定制：输入 JD + 岗位画像 → 匹配 → 生成两版简历。
3. 评测：输出 scorecard 与改进建议（含补证据任务）。
4. 投递：导出 PDF/DOCX 投递，记录渠道/版本/结果。
5. 迭代：基于追踪数据调权重、补证据、改模板，提高面试率。

## 5. 数据模型（核心资产）
### 5.1 Evidence Card（经历卡）
**硬规则：没有 `results` 或 `artifacts` 的卡不能进入简历候选池。**

字段（最小可用）：
- `id`：唯一标识
- `title`：战役名（一句话）
- `time_range`
- `context`：业务场景与约束（峰值/时效/多系统/不可延期）
- `role_scope`：职责边界（Owner/Tech Lead/执行）
- `actions`：3–5 条关键动作（方法论/架构/治理/排障/数据策略）
- `results`：至少 1 条量化结果（时延、成功率、事故率、成本、人天、交付周期等）
- `stack`：强相关技术栈
- `artifacts`：PR/监控截图/压测报告/复盘/设计文档（文件名或链接）
- `tags`：性能/稳定性/交付治理/DDD/数据治理/活动营销/门店等
- `interview_hooks`：面试深挖点与反问点

### 5.2 Job Profile（岗位画像）
- `target_role`
- `must_have`
- `nice_to_have`
- `keywords`（ATS/JD 关键词）
- `business_domain`
- `seniority_signal`（Owner/决策/带人/跨团队推进）
- `tone`（偏架构/偏交付）

### 5.3 Tracking（投递追踪）
- `company` / `role` / `channel` / `jd_source`
- `resume_version_id`（hash）
- `match_score_breakdown`
- `status`（投递/筛选/面试轮次/拒/offer）
- `feedback`（拒因/面试反馈/缺口）

## 6. 匹配与评分机制（可解释优先）
### 6.1 评分维度（建议默认权重，可配置）
- **关键词覆盖（K）**：技术栈词、职责词、领域词
- **业务域相似度（D）**：会员/营销/门店/电商等
- **级别信号（S）**：Owner、决策、跨团队推进、带人
- **量化强度（Q）**：指标数量与质量（含可验证性）
- **证据强度（E）**：artifacts 完整度与可信度
- **新近性（R）**：近 12–18 个月加权

输出：
- 总分 + 分项得分（可解释）
- Top N 经历卡（含理由）
- 缺口清单（缺关键词/缺能力证据/缺指标）
- 补证据任务（例如补压测结果、补监控截图、补复盘链接）

## 7. 简历生成策略（拼装优先，改写次之）
### 7.1 固定简历结构（1 页优先）
1) 顶部 10 秒区：定位 + 核心标签 + 技术栈
2) 核心亮点 3 条：结果 + 场景 + 动作
3) 工作经历：每段 4–6 bullets（动词 + 影响面 + 方法/约束 + 结果）
4) 代表项目 3 个：挑战-动作-结果（可深挖）
5) 开源/工具/内容（可选，产品化表述）

### 7.2 生成流程（强约束）
- 召回经历卡 → 排序 → 选 Top N → **模板拼装** → LLM 受控改写（仅措辞与关键词对齐，不新增事实）
- 输出两版：
  - **版本 A（架构/技术深度）**
  - **版本 B（交付治理/Owner）**
- 版本化：每份简历附 `resume_version_id`（hash），确保可追踪可回滚。

## 8. 质量评测（Scorecard）
### 8.1 指标
- must-have 关键词覆盖率（覆盖不足 → 给出建议插入的经历卡/证据点）
- 量化占比（含数字 bullets 比例）
- 空话率/重复度（泛词、同义重复、无主语无结果）
- 篇幅控制（1 页；段落长度阈值）
- 证据引用检查（关键表达是否可追溯到经历卡 artifacts）

### 8.2 输出
- Scorecard（总分 + 分项）
- 改进建议（按优先级）
- 补证据任务队列（可自动生成）

## 9. 技术架构（建议）
### 9.1 原则
- 核心地基确定性：**数据/索引/评分/拼装**必须可复现。
- “Pi 做底座”只适合作为**编排层/入口**，不是事实库与评分引擎的地基。

### 9.2 推荐实现（MVP）
- 数据层：Markdown/YAML（Git 版本管理）
- 提炼层：原始材料 → 证据卡（LLM 辅助抽取 + 质量门控）
- 索引层：BM25 全文检索 +（可选）Embedding +（可选）rerank
- 评分层：规则打分（可解释）+ 可配置权重
- 生成层：模板拼装 + LLM 受控改写
- 导出层：DOCX/PDF
- 追踪层：SQLite/CSV
- 交互层：CLI + 轻量 GUI（共享同一 pipeline）

### 9.3 目录结构建议
- `logs/`（原始材料，非事实源）
- `evidence_cards/`（经历卡）
- `job_profiles/`（岗位画像）
- `jd_inputs/`（JD 原文）
- `matching_reports/`（匹配报告）
- `outputs/`（简历版本）
- `tracking/`（投递追踪库）
- `templates/`（简历模板与 prompts）
- `tools/`（导入、校验、导出、评测 CLI）
- `ui/`（GUI 终版设计资产与后续桌面端实现）

## 10. 里程碑与交付物（按最短闭环）

- 拼装输出 + 受控改写

- Scorecard + 改进建议 + 补证据任务队列

- 投递记录、版本对比、权重调优建议
**交付物**：20 次投递的可分析数据集

## 11. 风险与对策
1) **日志地狱（材料多但无指标）**  
   - 对策：results/artifacts 门槛；缺失自动生成补证据任务  
2) **黑箱匹配不可优化**  
   - 对策：强制输出分项得分、理由、缺口清单  
3) **LLM 改写改坏事实**  
   - 对策：先拼装后改写；事实字段锁定；输出 diff 校验  
4) **自动投递合规风险**  
   - 对策：MVP 仅半自动（导出包/复制字段）；谨慎集成浏览器自动化

## 12. 竞品分析（项目竞品，含对比与策略）
> 竞品指：覆盖你价值链（匹配岗位→定制简历→投递/追踪）的产品与入口。

### 12.1 直接竞品（End-to-end 求职工作台）
#### Teal（Resume Builder + Match + Job Tracker）
- 定位为“一站式求职工作台”，含 Chrome 侧 job tracker 与简历分析/匹配能力。  
**你面临的压力**：用户心智强、“求职 OS”形态成熟。  
**你的反制**：把“事实源”升级为证据层（results+artifacts 硬约束）与“缺口→补证据任务”闭环（Teal 更偏简历/输入文本与工具集合）。  

#### Simplify Copilot（Autofill + Tailor + Auto Track）
- 主打浏览器端“自动填表、定制材料、自动追踪”，占据投递入口。  
**你面临的压力**：入口锁喉，用户在它那里完成“投递动作”。  
**你的反制**：MVP 不硬刚 autofill，先做“生成质量 + 可解释匹配 + 评测闭环”；后续用导出包/兼容格式对接或集成。

### 12.2 次直接竞品（覆盖你的一半价值）
#### Jobscan（Resume ↔ JD 扫描、Match Report、关键词优化）
- 强 ATS/关键词匹配与优化建议。  
**你如何区分**：从“关键词层匹配”升级到“证据层匹配”：每个关键词必须挂到某张经历卡的 results/artifacts，避免“写得像但没证据”。

#### Rezi（AI 简历生成 + ATS 检查/评分）
- 强 ATS 检查与关键词 targeting，提供评分与优化建议。  
**你如何区分**：Rezi 强在生成与 ATS 友好，你强在证据治理、可解释匹配、版本化 A/B 与追踪回流。

### 12.3 竞品对比表（与你项目最相关维度）
| 维度 | Teal | Simplify Copilot | Jobscan | Rezi | PiProofForge（本项目） |
|---|---|---|---|---|---|
| 事实源 | 简历/输入文本 | 表单/简历文本 | 简历文本 | 简历文本 | **证据层经历卡（results+artifacts 强约束）** |
| JD 匹配 | Match/分析 | 有（更轻） | 强（Match Report） | 有（Score/Checker） | **可解释分项得分 + 缺口任务** |
| 投递入口 | 间接 | 强（autofill） | 无 | 无 | **先半自动，后插件化** |
| 追踪 | 强（tracker） | 强（自动追踪） | 弱 | 弱 | **版本化 + 回流迭代** |
| 护城河类型 | 工作台 | 入口 | ATS/关键词 | ATS/模板/评分 | **证据资产 + 可解释匹配 + 评测闭环** |

### 12.4 竞品应对策略（写进路线图）
- P0：把“证据层门槛 + 可解释匹配 + 评分卡”做硬（这是你唯一能赢的地方）。  
- P1：追踪与版本化 A/B（把“迭代”做成系统优势）。  
- P2：谨慎接入投递入口（导出包/插件/对接第三方），避免早期陷入合规与维护泥潭。

## 13. 开源策略（建议）
### 13.1 开源有意义的部分（框架层）
- Evidence Card / Job Profile schema
- Evidence Extraction（从原始材料提炼结构化证据的流程与提示词）
- Matching + Evaluator（可解释评分与评测）
- CLI 脚手架与模板
- provider 插件接口（模型/embedding/导出器）

### 13.2 不开源的部分（护城河/风险）

## 14. 成功标准（验收）
- 10–20 张经历卡情况下，对新 JD 可在 15 分钟生成两版简历 + 匹配报告 + scorecard。
- 匹配报告能解释“为什么选这 3 个项目/经历”，并生成可执行缺口与补证据任务。
- 20 次投递后可用数据判断：哪类经历卡、哪种模板、哪种权重组合提升面试率。
